# Gitmoji コミットメッセージ作成

## ワークフロー

1. `git status` でステージ状態を確認する
   - **ステージ済みの変更がある** → それをそのまま対象にする（追加で `git add` しない）
   - **ステージ済みが無い** → `git add -A` で全変更（新規ファイル含む）をステージする。
     ただし `git status` にビルド成果物や一時ファイルなど明らかに無関係なものが含まれる場合は、
     ステージせずユーザーに確認する
2. `git diff --cached` でステージ済み変更の内容を確認する
3. 決定木に従って gitmoji を選択する
4. コミットメッセージを作成し `git commit` を実行する

## ユーザーに確認するケース

以下の場合は自分で判断せず、AskUserQuestion ツールでユーザーに確認する:

- **絵文字の選択に迷うとき** — 複数の gitmoji が当てはまりそうな場合
- **コミットを分割すべきとき** — 性質の異なる変更が混在している場合

ただし、変更の大半が1つのカテゴリに当てはまる場合は、そのカテゴリを優先する

## コミットメッセージのフォーマット

```
<emoji> <日本語での簡潔な説明>
```

1行のみ。詳細や本文は書かない。

## 決定木（優先順位順に評価）

以下の順序で判定し、最初に一致したものを使用する:

1. **🔥 ファイル・コードの削除** — `git diff --staged --name-status` が `D`
2. **🚚 ファイル移動・リネーム** — `git diff --staged --name-status` が `R`
3. **🤖 エージェント・プロンプト更新** — `.claude/`、`CLAUDE.md`、スキル定義の変更
4. **☁️ IaC 変更** — `*.tf`、`*.bicep` 等のインフラ定義ファイルの変更
5. **📝 ドキュメント更新** — `*.md`、`docs/`、コメントのみの変更（プログラム動作に影響しない）
6. **🔧 設定ファイル更新** — `*.json`、`*.yaml`、`*.toml`、CI/CD 設定（ロックファイルを除く）
7. **📦 依存関係の更新** — `*lock*`、`requirements.txt`、外部ライブラリの変更
8. **✅ テストの更新** — `test_*.py`、`*.test.ts`、`tests/*`（プロダクションコードの変更なし）
9. **✨ 新機能の実装** — 完全に新しい機能、新しい API エンドポイント
10. **♻️ リファクタリング** — 外部動作を変えないコード品質改善
11. **🐛 バグ修正** — 不正な動作の修正
12. **💄 見た目のみの変更** — UI・スタイル・レイアウトなど見た目の調整（ロジックの変更なし）
